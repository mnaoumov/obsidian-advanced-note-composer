# AGENTS.md

This file provides guidance to Claude Code (claude.ai/code) when working with code in this repository.

## Overview

Advanced Note Composer is an Obsidian plugin that enhances the built-in Note Composer core plugin, adding richer merge, split, extract, and swap operations for notes (with frontmatter merge strategies, footnote/backlink fixing, heading-aware merging, and templating). It is built on `obsidian-dev-utils`.

## Architecture

- **Root config files** are thin re-exports — actual logic lives in `scripts/` (`eslint.config.mts` → `scripts/eslint-config.ts`, etc.).
- **`src/`** — plugin source:
  - `main.ts` — Obsidian entry point (default export of `Plugin`); imports the global SCSS.
  - `plugin.ts` — `Plugin` class (extends `obsidian-dev-utils` `PluginBase`); `onloadImpl` wires up settings, settings tab, all command handlers, the Prism component, and release notes.
  - `plugin-settings.ts` — `PluginSettings` class plus the `Action`, `FrontmatterMergeStrategy`, `FrontmatterTitleMode`, and `TextAfterExtractionMode` enums; wraps `PathSettings` for include/exclude path filtering.
  - `plugin-settings-component.ts` — `PluginSettingsComponent` (settings load/save, value validators, legacy-settings converters).
  - `plugin-settings-tab.ts` — `PluginSettingsTab` settings UI (built with `SettingEx`/`SettingGroupEx`).
  - `prism-component.ts` — registers a Prism language for highlighting `{{token:format}}` template tokens.
  - `release-notes-component.ts` — shows version release notes once on layout ready.
  - `swapper.ts` — `swap` logic for exchanging two files or two folders (with optional full folder-structure swap).
  - `headings.ts` — helpers to extract a heading from editor selection/line.
  - `markdown-heading-document.ts` — parses markdown into a heading tree and merges documents heading-by-heading.
  - `insert-mode.ts` — `InsertMode` enum (append/prepend).
  - `move-selection-buffer.ts` — `MoveSelectionBuffer`, the transient holder for the "mark selection to move" feature (source note + captured selection + held source-note lock + the permanent notice); shared by the mark/move/cancel command handlers. `isOffsetInsideMarkedSelection` / `isRangeOverlappingMarkedSelection` reject an insert point/range that overlaps the marked text (a point is a zero-length range).
  - `mark-selection-to-move.ts` — `markSelectionToMove()`, the shared helper that locks the source note (blocking mutations), shows the marked-selection notice, and records the mark. The lock is taken with the dev-utils `lockForPath` `shouldReleaseOnAbort: true` + an `onUnlockRequested` callback that clears the buffer (identity-guarded on the mark's unique `abortController` so a stale controller can't cancel a later mark), so `Unlock active note` — which aborts every lock on the note, cancelling all operations that hold one — also releases the lock and tears the mark down (buffer cleared, notice hidden, highlight removed) without any hand-wired `abort` listener. Takes `shouldLockAllNotes`: when off it locks only the source note (`mode: 'file'`); when on it subtree-locks the vault root (`app.vault.getRoot().path`, `mode: 'subtree'`) so every note is read-only until the move completes or is cancelled. Used by the `Mark selection to move` command (passing the `shouldLockAllNotesWhenMarkingSelection` setting) and the split picker's "switch to smart cut".
  - `selection-highlight-component.ts` — `SelectionHighlightComponent`, persistently highlights the captured selection of a pending operation (a smart-cut mark, or a split/extract while its picker is open) in its source note. A CodeMirror 6 `StateField` (`selectionHighlightField`) + `setSelectionHighlightsEffect` (registered once via `plugin.registerEditorExtension`) renders the decorations; `addHighlight(file, ranges)` returns a `Disposable`, and `refresh()` (on add/remove + `active-leaf-change`) dispatches the effect to every markdown editor. The `@codemirror/*` imports are externalized by the dev-utils build, so they use Obsidian's runtime CM instance. Pure range logic (`computeHighlightRangesForFile`, `mergeHighlightRanges`, `buildSelectionHighlightDecorations`) is unit-tested; the workspace/editor glue (`refreshLeaf`) is `v8 ignore`d and confirmed via a desktop integration test.
  - `move-notice-component.ts` — `MoveNoticeComponent`, owns the permanent "Smart cut & paste" notice while a selection is marked. Builds it with up to four `ButtonComponent`s — move to top / bottom / at cursor (each `MoveMarkedSelectionEditorCommandHandlerBase.executeInActiveEditor()`), and Cancel move (`CancelMoveCommandHandler.cancelMove()`). Enabled state is refreshed via each move handler's `canExecuteInActiveEditor()` on `active-leaf-change` + `selectionchange`; the Cancel button is always enabled. The notice is non-dismissable — shown with `shouldHideOnClick: false` and `shouldShowCloseButton: false` (no `X`), so the only way to release the mark is the Cancel move button (or a programmatic `MoveSelectionBuffer.clear()`, which hides the notice); there is no `onHide`/close-confirmation auto-cancel (removed in issue-140, where clicking any button or the `X` was dismissing the notice before the action ran). **Configurable via the `Smart cut & paste` settings** (read from the injected `pluginSettingsComponent` at `showNotice()` time): each of the three move buttons is included only when its `shouldShowMove{ToTop,ToBottom,AtCursor}Button` flag is on (Cancel always shown), and when `shouldShowSmartCutNotice` is off `showNotice()` shows nothing and returns `null` (so `MoveSelectionBuffer` / `MarkedSelection.notice` are `Notice | null`, and `clear()` guards `notice?.hide()`). Defaults are all `true`, preserving the prior always-four-buttons behavior. Covered by `move-notice-component.test.ts` + the real-Obsidian `smart-cut-notice-settings.desktop.integration.test.ts`.
  - `filename-validation.ts` — regexes for invalid filename characters / trailing dots-or-spaces.
  - `templater.ts` — type augmentation for the optional Templater plugin API.
  - `command-handlers/` — one class per command (merge file/folder, swap file/folder, extract before/after cursor/current selection/this heading, split-by-headings), each extending an `obsidian-dev-utils` command-handler base. The "smart cut & paste" feature adds `mark-selection-to-move`, `move-marked-selection-here`, and the top/bottom variants, plus `cancel-move` (a `GlobalCommandHandler`). The `unlock-active-note` `GlobalCommandHandler` is **no longer registered by the plugin** — as of `obsidian-dev-utils` 86.0.0 `PluginBase` registers `UnlockActiveNoteCommandHandler` automatically (via its own `commandHandlerComponent`) before `onloadImpl` runs, so the plugin's command batch omits it. The six smart-cut move commands carry a **`Smart cut & paste:`** display-name prefix (their IDs are unprefixed and unchanged, so existing hotkeys survive). `mark-selection-to-move` (via the shared `markSelectionToMove()` helper) holds a `shouldBlockMutations` lock while the mark is pending: by default a `mode: 'file'` lock on the source note only; when the `shouldLockAllNotesWhenMarkingSelection` setting is on, a `mode: 'subtree'` lock on the vault root (`app.vault.getRoot().path`, i.e. `/`, which covers every path via `isChild`) so every note is read-only + mutation-blocked until the move completes or is cancelled. Either way it is one `Disposable` held in `MoveSelectionBuffer` and released via `clear()`. The helper takes the lock with `shouldReleaseOnAbort: true` + an `onUnlockRequested` callback (`() => MoveSelectionBuffer.clear()`, identity-guarded on the mark's `abortController` so a stale controller can't cancel a later mark), so **aborting the lock cancels the entire move** (dev-utils releases the lock on abort, and the callback drops the mark and hides the notice). The `Unlock active note` command is the **library** `UnlockActiveNoteCommandHandler` (constructed with just `{ app, resourceLockComponent }`), which `PluginBase` now registers itself (see above); it is available whenever the active note is covered by a lock — directly (a merge/split operation or the source-only mark lock) or via a `subtree`-locked ancestor (the all-notes mark's root lock) — checked by `resourceLockComponent.isLockedByAncestorForPath`, and on execute calls `resourceLockComponent.requestUnlockForPath(activeFile)`, which resolves the covering owner (the path itself or its `subtree`-locked ancestor), aborts its controller(s) → for a mark that fires `onUnlockRequested` → cancel, and releases the lock. The lock indicator's right-click unlock hits the same abort path, so it also cancels the whole move. Confirmed against real Obsidian by `unlock-active-note.desktop.integration.test.ts` (the source-note write blocked while marked succeeds again after the unlock). The move-into-active-note commands share `MoveMarkedSelectionEditorCommandHandlerBase` (validation, mtime guard, buffer clear, `SplitComposer` handoff); subclasses only supply the insert point via `resolveInsertion()` and, optionally, override `resolveOptions()`. The base also exposes public `canExecuteInActiveEditor()` / `executeInActiveEditor()` (resolve the active `MarkdownView`, delegate to `canExecuteEditor`/`executeEditor`) so the notice buttons can run each command against the active editor. `MoveMarkedSelectionHereEditorCommandHandler` (registered twice via an `isAdvanced` flag for the default and advanced-options commands) keeps the paste-options modal and, via `resolveInsertion()`, returns the target's selection **range** (`getCursor('from')`/`('to')`) so an active target selection is *replaced* (paste-over-selection); with no selection the range collapses to the cursor. `MoveMarkedSelectionToEdgeEditorCommandHandler` (registered once per `InsertMode` → `move-marked-selection-to-top-of-file` / `move-marked-selection-to-bottom-of-file`, no default hotkeys) returns a `null` cursor offset so the composer derives top/bottom from `insertMode`. `CancelMoveCommandHandler` exposes a public `cancelMove()` (used by both its `execute()` and the notice's Cancel button). The move token is created by the shared `createMoveToken()` (`src/move-token.ts`).
  - **Smart cut & paste template.** `PluginSettings.smartCutAndPasteTemplate` (`''` default) is the template applied to *all* smart cut & paste moves (mark → move here / at cursor / to top / to bottom), so a smart-cut paste can be formatted differently from an ordinary split-to-new-file extract. `SplitComposer` carries an `isSmartCutAndPasteMove` flag (set `true` by `MoveMarkedSelectionEditorCommandHandlerBase.executeEditor`, `false`/absent for ordinary `Extract …`/split, including same-note extract from the picker). `SplitComposer.getTemplate` returns `smartCutAndPasteTemplate` first when the flag is on and it is non-empty; when empty it falls through to the existing split → merge resolution (the documented fallback chain). Validated in `plugin-settings-component.ts` (must contain `{{content}}` when non-empty, mirroring `splitTemplate`); shown in the `Smart cut & paste` settings group via `addCodeHighlighter`. It is a global setting only — not per-move overridable in the advanced paste-options modal. Confirmed against real Obsidian by `smart-cut-template.desktop.integration.test.ts` (a move uses the smart-cut template; emptying it falls back to the split template).
  - `composers/` — core merge/split engine: `ComposerBase` (frontmatter merge, footnote/backlink/link fixing, templating) with `MergeComposer` and `SplitComposer` subclasses. The exported `resolveInsertOffset(content, insertMode)` maps an `InsertMode` to a concrete offset (Append = end of note, Prepend = just after frontmatter) and backs both `insertContent`'s append/prepend path and the move flow's top/bottom offset. `ComposerBase` also supports a move mode via an `insertToken` (content replaces the token at the insert point instead of appending/prepending); `SplitComposer.insertTokenIntoTargetFile` derives that offset from `insertMode` when `targetCursorOffset` is `null` (top/bottom), or uses the pinned cursor offset otherwise, and re-maps captured offsets for same-note moves. The token **replaces the `[targetCursorOffset, targetCursorEndOffset]` range** (paste-over-selection when the target has a selection); with no selection the range is empty (`targetCursorEndOffset` defaults to the start), so it is a plain insertion. The same-note offset re-map shifts captured selections by `delta = insertToken.length - (endOffset - startOffset)`. **Same-note split IS a move:** `SplitComposer`'s constructor synthesizes an `insertToken` when `sourceFile === targetFile` (so a same-note extract routes through the same-note-move ordering instead of the broken append/prepend path that would collapse the editor selection and turn the move into a copy), and forces `shouldFixFootnotes`/`shouldIncludeFrontmatter` off for same-note (footnote-fixing would rename+dangle the moved ref; re-including the note's own frontmatter is meaningless). The residual left in the source (link/embed/nothing) is resolved via `textAfterExtractionMode` (a `SplitComposer` param overriding the `Text after extraction` setting; defaults a same-note move to `None` unless the `shouldApplyTextAfterExtractionToSameFile` setting is on) in `replaceSourceSelection`. **Same-note move ordering:** for a same-note move (`isSameNoteMove`) the source selection is removed *before* the target write — the same-file write collapses the editor selection, so a post-write `replaceSelection` would be a no-op (turning the move into a copy); the write reads the post-removal buffer, so the removal survives, and footnote defs need no cleanup (refs + defs stay in the note). Cross-note/split keeps insert-then-remove so `fixFootnotes`/`updateEditorSelections` can extend the selection to also drop orphaned footnote definitions. **Data-loss guard:** `insertTokenIntoTargetFile` aborts (with a notice) when the derived/pinned insert offset falls strictly inside a captured selection (e.g. a same-note "top" move of a selection spanning the frontmatter boundary) — the token would otherwise be deleted with the source; the top/bottom command handlers also gate this in `canExecute` via `MoveSelectionBuffer.isOffsetInsideMarkedSelection`.
  - `item-selectors/` — resolve a chosen suggestion into a concrete target file: `ItemSelectorBase`, `MergeItemSelector`, `SplitItemSelector`.
  - `modals/` — suggestion/confirmation UI: `SuggestModalBase` and the merge/split/swap file/folder modals, plus `paste-options-modal.ts` (the advanced-move options modal + `MoveOptions`). The instruction-bar builder (`SuggestModalCommandBuilder`) now comes from `obsidian-dev-utils/obsidian/modals/suggest-modal-command-builder`. `SuggestModalBase` excludes the source note from its suggestions unless the `shouldAllowSameFile` flag is set; only `SplitFileModal` sets it (so `Extract …` can target the same note — `Enter` = bottom, `Shift+Enter` = top), keeping merge/swap from offering a file against itself. `SplitFileModal` also offers a **"Switch to smart cut & paste"** button — always rendered in `onOpen`, but disabled when `!canSwitchToSmartCut` (the caller did not wire the buffer + notice) — plus an `Alt+S` shortcut registered only when `canSwitchToSmartCut`. Both call `switchToSmartCut()`, which resolves the modal with a bare `switch-to-smart-cut` result (no target), and `prepareForSplitFile` handles it by calling `markSelectionToMove()` and staying on the source note instead of splitting (returning `null` so the extract handler does nothing further) — it does **not** open the picker's merely-highlighted suggestion, which would switch the active note the user never chose (issue #141). (The confirmation dialog's own "Switch to smart cut & paste", by contrast, opens the already-confirmed target.) The result type is a discriminated union (`action: 'split' | 'switch-to-smart-cut'`); the four `Extract …` handlers thread `moveSelectionBuffer` + `moveNoticeComponent` + `selectionHighlightComponent` into `prepareForSplitFile` to enable it. The `ConfirmDialogModal` is now a **shared, parameterized modal** in `confirm-dialog-modal.ts` (see below) used by the split, merge-file, and merge-folder confirmation flows.
  - `confirm-dialog-modal.ts` — the shared `ConfirmDialogModal` (v8-ignored UI) + `ConfirmDialogModalResult` + params. It is generic: the caller passes `title`, `confirmButtonText`/`confirmButtonMobileText`, a `buildContent(fragment)` async callback for the body (so each flow renders its own question/links/extras — split adds the "Source content to split" section, merge renders the "will be deleted" body), `canReselectTarget`, and an optional `switchToSmartCut: { canSwitch }` (present **only** for split — when omitted the "Switch to smart cut & paste" button/`Alt+S` are not rendered at all). It renders: the body; a "Don't ask again" checkbox (desktop) / "\<verb\> and don't ask again" button (mobile); a **"Change target"** button + `Alt+C` (disabled when `!canReselectTarget`) resolving `shouldReselectTarget: true`; the optional smart-cut button + `Alt+S` resolving `shouldSwitchToSmartCut: true`; the primary confirm (`Enter`, reads modifiers for `insertMode`); and Cancel (`Escape`). The result's generic `shouldAskAgain` is mapped back to each flow's own `shouldAskBefore*` setting by its `prepareFor*`/`selectTarget*`.
  - **Reselect ("Change target") loops.** Each target-choosing flow wraps its picker + confirmation in a `for (;;)` loop; on `shouldReselectTarget` it cleans up and re-runs, else it exits on confirm/cancel/(split: switch-to-smart-cut). `prepareForSplitFile` (`confirmSplit()` helper holds the `using _targetLock` around the dialog) and `prepareForMergeFile` (`confirmMerge()` helper, same lock pattern) both **trash the abandoned target when `isNewTargetFile`** and re-seed the reopened picker with the previous query via `SuggestModalBase.initialInputValue` (lifted into the base's `onOpen`, which seeds `inputEl.value` + `updateSuggestions()`; used by the split & merge-file `SuggestModalBase` pickers). `selectTargetFolderForMergeFolder` has the same loop but simpler — no target lock and no trash (folders are existing), and no preselect (its picker is a `FuzzySuggestModal`, not `SuggestModalBase`). Only split passes `switchToSmartCut`; merge only offers Change target + confirm/cancel. Confirmed against real Obsidian by `confirm-dialog-change-target{,-merge-file,-merge-folder}.desktop.integration.test.ts` (change target reopens the picker and completes the operation into the newly chosen target, leaving the abandoned one untouched).
  - `open-minimizable-modal.ts` — two openers. `openMinimizableModal(modal, abortController?)` wraps a modal in the dev-utils `MinimizableModal` so it can be minimized to a floating bar while its operation's lock is held; `openModal(modal, abortController?)` opens the modal plainly, with **no** minimize button. **Minimizable**: every confirmation dialog (split / merge file / merge folder) **and the split/extract picker** — the latter is deliberately minimizable so a pending extract can be parked and cancelled from the minimized bar (issue #130). **Plain (`openModal`)**: the other initial pickers — merge file/folder and swap file/folder — because a target has not been chosen yet, so minimizing serves no purpose and risks the user forgetting which note the operation was triggered on (issue #125). Both openers wire an optional `abortController` `signal → modal.close()` so an external unlock (the lock indicator's Unlock menu / `Unlock active note` command) closes the open (possibly minimized) modal and cancels the setup flow. The minimized bar's **Cancel button is provided by dev-utils and is on by default** (`obsidian-dev-utils` ≥ 86.0.0 defaults `MinimizableModalConstructorOptions.shouldShowCancelButton` to `true`), so the wrapper passes **no** option — every minimizable modal gets a bar `Cancel` (✕) that closes it. For a lock-holding flow (extract/split, merge) closing releases the lock via the `onClose` → `promiseResolve(null)` → `using` chain, so the bar Cancel cancels the whole operation and unlocks the note — the discoverable alternative to the Unlock command that issue #130 asked for (while minimized, the peek-lock suppresses `contextmenu`, hiding the lock-indicator's right-click Unlock menu). Confirmed against real Obsidian by `unlock-cancel-minimized-modal.desktop.integration.test.ts` (minimize the extract picker, click the bar Cancel → modal gone AND the source lock indicator gone) and, for the plain pickers, `minimize-button.desktop.integration.test.ts` (the initial merge picker renders no minimize button).
  - `styles/` — `main.scss` plus the SCSS module type declaration.
- **Smart-cut reverse flow.** A marked selection can switch back into the split/extract flow through the
  permanent notice or the `open-split-modal` command; `OpenSplitModalCommandHandler` clears the mark,
  restores the source-editor selection, then delegates to the normal extract-current-selection handler.
- **`main` field** points to `src/main.ts` (Obsidian plugin source entry; built artifact is `dist/build/main.js`, not published to npm).

## Demo vault

- **`demo-vault/`** is a self-contained Obsidian vault demoing every headline feature (merge/split/extract/swap/smart-cut). It ships a lean `.obsidian/` skeleton — `app.json`, `community-plugins.json` = `["demo-vault-helper", "advanced-note-composer"]`, and a baked `plugins/advanced-note-composer/data.json` for the plugin under test (no `appearance.json`/`core-plugins.json`; nothing about CodeScript Toolkit is committed — the helper installs and configures it at runtime). The plugin's own build is **not** committed — `main.js`/`manifest.json`/`styles.css` are gitignored and injected at release by ODU's `archivePluginDemoVault` (inherited via the `version` script; runs automatically once `demo-vault/` exists, zipping to `dist/build/demo-vault-<version>.zip`). At runtime the `advanced-note-composer:open-demo-vault` command (`OpenDemoVaultCommandHandler`, registered in `plugin.ts`'s command batch) downloads and opens that asset.
- **Bootstrap = a committed, plugin-agnostic `demo-vault-helper` plugin.** `plugins/demo-vault-helper/{manifest.json,main.js}` are the ONE plugin whose build **is** committed (re-included in `demo-vault/.gitignore` against the root `main.js` rule; hand-written CommonJS, no build step). It is **universal** — intended to be byte-identical across every plugin's demo vault, carrying nothing plugin-specific: on layout-ready it installs + enables CodeScript Toolkit (`fix-require-modules`) — replicating ODU's `installCommunityPlugin` flow (`app.plugins.installPlugin`/`enablePluginAndSave` + `requestUrl` to the community registry) because it CANNOT import ODU (reachable only through CST's `require`, which it bootstraps) — points CST's `modulesRoot` at `_assets/CodeScriptToolkit` (writing CST's `data.json` at runtime, so no CST config is committed), and then runs the vault's own startup script via `window.require('//_assets/CodeScriptToolkit/startup.ts')` (the `//` is vault-root-relative, so the path is identical in every demo vault regardless of CST's `modulesRoot`). No app reload (enabling loads CST live). This removes the manual CST install step. (Candidate to be centralized in ODU rather than copy-pasted per vault — see S2.)
- **Per-vault startup lives in `_assets/CodeScriptToolkit/startup.ts`** — a CST module the helper `require`s; it opens `00 Start.md` (and is where any plugin-specific startup setup would go), keeping the helper itself generic.
- **Interactive note buttons still use CodeScript Toolkit code-buttons.** `_assets/CodeScriptToolkit/` is CST's `modulesRoot`; `demoSetup.ts` (`installAndEnable` / `changeSettingsAndReload` / `bindHotkey`) backs the notes' "change a setting and reload" buttons. Every button note also lists the manual equivalent, so the vault works even without CST.
- **Gate wiring for the vault.** ESLint and `tsc` are include-based, so they ignore `demo-vault/`. A root `dprint.json` (extends the ODU template, adds `demo-vault` to `excludes`) keeps dprint off it. cspell + markdownlint + linkinator **do** scan the notes: `demo-vault/.markdownlint-cli2.jsonc` disables `MD041`/`MD052`, demo words live in `cspell.json`, and intra-vault links to spaced filenames use angle-bracket markdown (`[t](<./Spaced name.md>)`) since wikilinks aren't linkinator-checked. Requires `obsidian-dev-utils >= 87.0.2` (the linkinator spaced-filename fix).

## Resource locking & transactional rollback

Every merge/split/swap operation (files **and** folders) locks the resources it touches
(`ResourceLockComponent.lockForPath({ shouldBlockMutations: true })`) against edit/delete/rename/move,
detects external changes and aborts, and runs its vault mutations inside a reversible dev-utils
`VaultTransaction` that commits on success and rolls back on cancel/error. The shared runner is
`src/locked-transaction.ts` (`runLockedTransaction`), used by the composers and the swap/merge-folder
handlers; folder-merge threads one spanning transaction into each `MergeComposer`. Requires
`obsidian-dev-utils` ≥ 84.1.0 (`tx.rollback` uses `syncOpenEditorBuffersForPath` so split's rollback
survives an open editor). Unit tests use the real bridge (`App.createConfigured__()` + real
`ResourceLockComponent`/`VaultTransaction`), 100% coverage.

The former integration-only branches are now fully unit-covered against `obsidian-test-mocks` ≥ 3.5.1,
so their `/* v8 ignore */`s are gone: the nested / differently-named folder-swap paths in `swapper.ts`
(3.5.0 made folder rename cascade to descendants and `getAvailablePath` de-duplicate), and the
backlink-rewrite `linkConverter` in `merge-composer.ts` (`fixBacklinks`) — 3.5.0 gave synchronous link
indexing and 3.5.1 fixed the markdown parser's link end offset to the exclusive `start + length`, so
dev-utils' `editLinks` write path now completes against the mock.
