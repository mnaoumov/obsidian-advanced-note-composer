## Overview

Advanced Note Composer is an Obsidian plugin that enhances the built-in Note Composer core plugin, adding richer merge, split, extract, and swap operations for notes (with frontmatter merge strategies, footnote/backlink fixing, heading-aware merging, and templating). It is built on `obsidian-dev-utils`.

## Architecture

- **Root config files** are thin re-exports — actual logic lives in `scripts/` (`eslint.config.mts` → `scripts/eslint-config.ts`, etc.).
- **`src/`** — plugin source:
  - `main.ts` — Obsidian entry point (default export of `Plugin`); imports the global SCSS.
  - `plugin.ts` — `Plugin` class (extends `obsidian-dev-utils` `PluginBase`); `onloadImpl` wires up settings, settings tab, all command handlers, the Prism component, and release notes.
  - `plugin-settings.ts` — `PluginSettings` class plus the `Action`, `FrontmatterMergeStrategy`, `FrontmatterTitleMode`, and `TextAfterExtractionMode` enums; wraps `PathSettings` for include/exclude path filtering.
  - `plugin-settings-component.ts` — `PluginSettingsComponent` (settings load/save, value validators, legacy-settings converters).
  - `plugin-settings-tab.ts` — `PluginSettingsTab` settings UI (built with `SettingEx`/`SettingGroupEx`).
  - `prism-component.ts` — registers a Prism language for highlighting `{{token:format}}` template tokens.
  - `release-notes-component.ts` — shows version release notes once on layout ready.
  - `swapper.ts` — `swap` logic for exchanging two files or two folders (with optional full folder-structure swap).
  - `headings.ts` — helpers to extract a heading from editor selection/line.
  - `markdown-heading-document.ts` — parses markdown into a heading tree and merges documents heading-by-heading.
  - `insert-mode.ts` — `InsertMode` enum (append/prepend).
  - `move-selection-buffer.ts` — `MoveSelectionBuffer`, the transient holder for the "mark selection to move" feature (source note + captured selection + held source-note lock + the permanent notice); shared by the mark/move/cancel command handlers. `isOffsetInsideMarkedSelection` / `isRangeOverlappingMarkedSelection` reject an insert point/range that overlaps the marked text (a point is a zero-length range).
  - `mark-selection-to-move.ts` — `markSelectionToMove()`, the shared helper that locks the source note (blocking mutations), shows the marked-selection notice, and records the mark. The lock is taken with the dev-utils `lockForPath` `shouldReleaseOnAbort: true` + an `onUnlockRequested` callback that clears the buffer (identity-guarded on the mark's unique `abortController` so a stale controller can't cancel a later mark), so `Unlock active note` — which aborts every lock on the note, cancelling all operations that hold one — also releases the lock and tears the mark down (buffer cleared, notice hidden, highlight removed) without any hand-wired `abort` listener. Takes `shouldLockAllNotes`: when off it locks only the source note (`mode: 'file'`); when on it subtree-locks the vault root (`app.vault.getRoot().path`, `mode: 'subtree'`) so every note is read-only until the move completes or is cancelled. Used by the `Mark selection to move` command (passing the `shouldLockAllNotesWhenMarkingSelection` setting) and the split picker's "switch to smart cut".
  - `selection-highlight-component.ts` — `SelectionHighlightComponent`, persistently highlights the captured selection of a pending operation (a smart-cut mark, or a split/extract while its picker is open) in its source note. A CodeMirror 6 `StateField` (`selectionHighlightField`) + `setSelectionHighlightsEffect` (registered once via `plugin.registerEditorExtension`) renders the decorations; `addHighlight(file, ranges)` returns a `Disposable`, and `refresh()` (on add/remove + `active-leaf-change`) dispatches the effect to every markdown editor. The `@codemirror/*` imports are externalized by the dev-utils build, so they use Obsidian's runtime CM instance. Pure range logic (`computeHighlightRangesForFile`, `mergeHighlightRanges`, `buildSelectionHighlightDecorations`) is unit-tested; the workspace/editor glue (`refreshLeaf`) is `v8 ignore`d and confirmed via a desktop integration test.
  - `move-notice-component.ts` — `MoveNoticeComponent`, owns the permanent "Smart cut & paste" notice while a selection is marked. Builds it with up to four `ButtonComponent`s — move to top / bottom / at cursor (each `MoveMarkedSelectionEditorCommandHandlerBase.executeInActiveEditor()`), and Cancel move (`CancelMoveCommandHandler.cancelMove()`). Enabled state is refreshed via each move handler's `canExecuteInActiveEditor()` on `active-leaf-change` + `selectionchange`; the Cancel button is always enabled. **Configurable via the `Smart cut & paste` settings** (read from the injected `pluginSettingsComponent` at `showNotice()` time): each of the three move buttons is included only when its `shouldShowMove{ToTop,ToBottom,AtCursor}Button` flag is on (Cancel always shown), and when `shouldShowSmartCutNotice` is off `showNotice()` shows nothing and returns `null` (so `MoveSelectionBuffer` / `MarkedSelection.notice` are `Notice | null`, and `clear()` guards `notice?.hide()`). Defaults are all `true`, preserving the prior always-four-buttons behavior. Covered by `move-notice-component.test.ts` + the real-Obsidian `smart-cut-notice-settings.desktop.integration.test.ts`.
  - `filename-validation.ts` — regexes for invalid filename characters / trailing dots-or-spaces.
  - `templater.ts` — type augmentation for the optional Templater plugin API.
  - `command-handlers/` — one class per command (merge file/folder, swap file/folder, extract before/after cursor/current selection/this heading, split-by-headings), each extending an `obsidian-dev-utils` command-handler base. The "smart cut & paste" feature adds `mark-selection-to-move`, `move-marked-selection-here`, and the top/bottom variants, plus `cancel-move` and `unlock-active-note` (both `GlobalCommandHandler`s). The six smart-cut move commands carry a **`Smart cut & paste:`** display-name prefix (their IDs are unprefixed and unchanged, so existing hotkeys survive). `mark-selection-to-move` (via the shared `markSelectionToMove()` helper) holds a `shouldBlockMutations` lock while the mark is pending: by default a `mode: 'file'` lock on the source note only; when the `shouldLockAllNotesWhenMarkingSelection` setting is on, a `mode: 'subtree'` lock on the vault root (`app.vault.getRoot().path`, i.e. `/`, which covers every path via `isChild`) so every note is read-only + mutation-blocked until the move completes or is cancelled. Either way it is one `Disposable` held in `MoveSelectionBuffer` and released via `clear()`. The helper takes the lock with `shouldReleaseOnAbort: true` + an `onUnlockRequested` callback (`() => MoveSelectionBuffer.clear()`, identity-guarded on the mark's `abortController` so a stale controller can't cancel a later mark), so **aborting the lock cancels the entire move** (dev-utils releases the lock on abort, and the callback drops the mark and hides the notice). The `Unlock active note` command is the **library** `UnlockActiveNoteCommandHandler` (imported from `obsidian-dev-utils/obsidian/command-handlers/unlock-active-note-command-handler`, constructed with just `{ app, resourceLockComponent }`); it is available whenever the active note is covered by a lock — directly (a merge/split operation or the source-only mark lock) or via a `subtree`-locked ancestor (the all-notes mark's root lock) — checked by `resourceLockComponent.isLockedByAncestorForPath`, and on execute calls `resourceLockComponent.requestUnlockForPath(activeFile)`, which resolves the covering owner (the path itself or its `subtree`-locked ancestor), aborts its controller(s) → for a mark that fires `onUnlockRequested` → cancel, and releases the lock. The lock indicator's right-click unlock hits the same abort path, so it also cancels the whole move. Confirmed against real Obsidian by `unlock-active-note.desktop.integration.test.ts` (the source-note write blocked while marked succeeds again after the unlock). The move-into-active-note commands share `MoveMarkedSelectionEditorCommandHandlerBase` (validation, mtime guard, buffer clear, `SplitComposer` handoff); subclasses only supply the insert point via `resolveInsertion()` and, optionally, override `resolveOptions()`. The base also exposes public `canExecuteInActiveEditor()` / `executeInActiveEditor()` (resolve the active `MarkdownView`, delegate to `canExecuteEditor`/`executeEditor`) so the notice buttons can run each command against the active editor. `MoveMarkedSelectionHereEditorCommandHandler` (registered twice via an `isAdvanced` flag for the default and advanced-options commands) keeps the paste-options modal and, via `resolveInsertion()`, returns the target's selection **range** (`getCursor('from')`/`('to')`) so an active target selection is *replaced* (paste-over-selection); with no selection the range collapses to the cursor. `MoveMarkedSelectionToEdgeEditorCommandHandler` (registered once per `InsertMode` → `move-marked-selection-to-top-of-file` / `move-marked-selection-to-bottom-of-file`, no default hotkeys) returns a `null` cursor offset so the composer derives top/bottom from `insertMode`. `CancelMoveCommandHandler` exposes a public `cancelMove()` (used by both its `execute()` and the notice's Cancel button). The move token is created by the shared `createMoveToken()` (`src/move-token.ts`).
  - **Smart cut & paste template.** `PluginSettings.smartCutAndPasteTemplate` (`''` default) is the template applied to *all* smart cut & paste moves (mark → move here / at cursor / to top / to bottom), so a smart-cut paste can be formatted differently from an ordinary split-to-new-file extract. `SplitComposer` carries an `isSmartCutAndPasteMove` flag (set `true` by `MoveMarkedSelectionEditorCommandHandlerBase.executeEditor`, `false`/absent for ordinary `Extract …`/split, including same-note extract from the picker). `SplitComposer.getTemplate` returns `smartCutAndPasteTemplate` first when the flag is on and it is non-empty; when empty it falls through to the existing split → merge resolution (the documented fallback chain). Validated in `plugin-settings-component.ts` (must contain `{{content}}` when non-empty, mirroring `splitTemplate`); shown in the `Smart cut & paste` settings group via `addCodeHighlighter`. It is a global setting only — not per-move overridable in the advanced paste-options modal. Confirmed against real Obsidian by `smart-cut-template.desktop.integration.test.ts` (a move uses the smart-cut template; emptying it falls back to the split template).
  - `composers/` — core merge/split engine: `ComposerBase` (frontmatter merge, footnote/backlink/link fixing, templating) with `MergeComposer` and `SplitComposer` subclasses. The exported `resolveInsertOffset(content, insertMode)` maps an `InsertMode` to a concrete offset (Append = end of note, Prepend = just after frontmatter) and backs both `insertContent`'s append/prepend path and the move flow's top/bottom offset. `ComposerBase` also supports a move mode via an `insertToken` (content replaces the token at the insert point instead of appending/prepending); `SplitComposer.insertTokenIntoTargetFile` derives that offset from `insertMode` when `targetCursorOffset` is `null` (top/bottom), or uses the pinned cursor offset otherwise, and re-maps captured offsets for same-note moves. The token **replaces the `[targetCursorOffset, targetCursorEndOffset]` range** (paste-over-selection when the target has a selection); with no selection the range is empty (`targetCursorEndOffset` defaults to the start), so it is a plain insertion. The same-note offset re-map shifts captured selections by `delta = insertToken.length - (endOffset - startOffset)`. **Same-note split IS a move:** `SplitComposer`'s constructor synthesizes an `insertToken` when `sourceFile === targetFile` (so a same-note extract routes through the same-note-move ordering instead of the broken append/prepend path that would collapse the editor selection and turn the move into a copy), and forces `shouldFixFootnotes`/`shouldIncludeFrontmatter` off for same-note (footnote-fixing would rename+dangle the moved ref; re-including the note's own frontmatter is meaningless). The residual left in the source (link/embed/nothing) is resolved via `textAfterExtractionMode` (a `SplitComposer` param overriding the `Text after extraction` setting; defaults a same-note move to `None` unless the `shouldApplyTextAfterExtractionToSameFile` setting is on) in `replaceSourceSelection`. **Same-note move ordering:** for a same-note move (`isSameNoteMove`) the source selection is removed *before* the target write — the same-file write collapses the editor selection, so a post-write `replaceSelection` would be a no-op (turning the move into a copy); the write reads the post-removal buffer, so the removal survives, and footnote defs need no cleanup (refs + defs stay in the note). Cross-note/split keeps insert-then-remove so `fixFootnotes`/`updateEditorSelections` can extend the selection to also drop orphaned footnote definitions. **Data-loss guard:** `insertTokenIntoTargetFile` aborts (with a notice) when the derived/pinned insert offset falls strictly inside a captured selection (e.g. a same-note "top" move of a selection spanning the frontmatter boundary) — the token would otherwise be deleted with the source; the top/bottom command handlers also gate this in `canExecute` via `MoveSelectionBuffer.isOffsetInsideMarkedSelection`.
  - `item-selectors/` — resolve a chosen suggestion into a concrete target file: `ItemSelectorBase`, `MergeItemSelector`, `SplitItemSelector`.
  - `modals/` — suggestion/confirmation UI: `SuggestModalBase` and the merge/split/swap file/folder modals, plus `paste-options-modal.ts` (the advanced-move options modal + `MoveOptions`). The instruction-bar builder (`SuggestModalCommandBuilder`) now comes from `obsidian-dev-utils/obsidian/modals/suggest-modal-command-builder`. `SuggestModalBase` excludes the source note from its suggestions unless the `shouldAllowSameFile` flag is set; only `SplitFileModal` sets it (so `Extract …` can target the same note — `Enter` = bottom, `Shift+Enter` = top), keeping merge/swap from offering a file against itself. `SplitFileModal` also offers a **"Switch to smart cut & paste"** button — always rendered in `onOpen`, but disabled when `!canSwitchToSmartCut` (the caller did not wire the buffer + notice) — plus an `Alt+S` shortcut registered only when `canSwitchToSmartCut`. Both call `switchToSmartCut()`, which resolves the modal with a `switch-to-smart-cut` result carrying the highlighted target file, and `prepareForSplitFile` handles it by calling `markSelectionToMove()` and opening that note instead of splitting (returning `null` so the extract handler does nothing further). The result type is a discriminated union (`action: 'split' | 'switch-to-smart-cut'`); the four `Extract …` handlers thread `moveSelectionBuffer` + `moveNoticeComponent` + `selectionHighlightComponent` into `prepareForSplitFile` to enable it. The `ConfirmDialogModal` (shown when `Ask before splitting` is on) carries the **same "Switch to smart cut & paste" button** (disabled when `!canSwitchToSmartCut`), resolving with `shouldSwitchToSmartCut: true`; `prepareForSplitFile` then marks + opens the already-resolved target instead of splitting. The `ConfirmDialogModal` also carries a **"Change target"** button (disabled when `!canReselectTarget`, i.e. a heading-driven `shouldSkipModal` split with no picker to reopen) plus an `Alt+C` shortcut, resolving with `shouldReselectTarget: true`. To support it, `prepareForSplitFile` wraps its target-selection + confirmation in a `for (;;)` loop: on `shouldReselectTarget` it trashes the abandoned target (when `isNewTargetFile`), seeds the picker with the previous query (`pickerSeed` → `SplitFileModal.initialInputValue`, which seeds the input in place of the heading), and re-runs; it exits the loop on confirm/cancel/switch-to-smart-cut. The confirmation dialog + its target lock live in the module-level `confirmSplit()` helper (the `using _targetLock` releases when it returns — before the split re-locks to do the work, and before the picker reopens on reselect). Confirmed against real Obsidian by `confirm-dialog-change-target.desktop.integration.test.ts` (change target reopens the picker and splits into the newly chosen note, leaving the abandoned target untouched).
  - `styles/` — `main.scss` plus the SCSS module type declaration.
- **`main` field** points to `src/main.ts` (Obsidian plugin source entry; built artifact is `dist/build/main.js`, not published to npm).

## Resource locking & transactional rollback

Every merge/split/swap operation (files **and** folders) locks the resources it touches
(`ResourceLockComponent.lockForPath({ shouldBlockMutations: true })`) against edit/delete/rename/move,
detects external changes and aborts, and runs its vault mutations inside a reversible dev-utils
`VaultTransaction` that commits on success and rolls back on cancel/error. The shared runner is
`src/locked-transaction.ts` (`runLockedTransaction`), used by the composers and the swap/merge-folder
handlers; folder-merge threads one spanning transaction into each `MergeComposer`. Requires
`obsidian-dev-utils` ≥ 84.1.0 (`tx.rollback` uses `syncOpenEditorBuffersForPath` so split's rollback
survives an open editor). Unit tests use the real bridge (`App.createConfigured__()` + real
`ResourceLockComponent`/`VaultTransaction`), 100% coverage.

The former integration-only branches are now fully unit-covered against `obsidian-test-mocks` ≥ 3.5.1,
so their `/* v8 ignore */`s are gone: the nested / differently-named folder-swap paths in `swapper.ts`
(3.5.0 made folder rename cascade to descendants and `getAvailablePath` de-duplicate), and the
backlink-rewrite `linkConverter` in `merge-composer.ts` (`fixBacklinks`) — 3.5.0 gave synchronous link
indexing and 3.5.1 fixed the markdown parser's link end offset to the exclusive `start + length`, so
dev-utils' `editLinks` write path now completes against the mock.
